程式碼解說

閱讀順序：models → activities → workflow → worker → starter
  1. models.py：定義改價規格與審核決定。
  2. activities.py：通知、記錄決定、套用新價格。
  3. workflow.py：ProductChangeWorkflow.run 等待 Update 或 timeout。
  4. worker.py：註冊審批 Workflow 與三個 Activity。
  5. starter.py：啟動審批流程；client_update.py 負責送入審核決定。

activities.py — Activity 清單

Activity用途

apply_new_price核准後套用新的商品價格。

workflow.py — Update + Wait Condition

@workflow.defn
class ProductChangeWorkflow:
    def __init__(self):
        self._decision: ReviewDecision | None = None  # 審核決定，初始為空

    @workflow.run
    async def run(self, spec: ProductChangeSpec) -> None:
        await workflow.execute_activity(notify_reviewer, spec, **_ACT_CFG)

        try:
            # 等待條件成立（有人送了審核決定），或 48 小時後 timeout
            await workflow.wait_condition(
                lambda: self._decision is not None,
                timeout=timedelta(hours=48),
            )
        except TimeoutError:
            # 逾時：系統自動拒絕
            self._decision = ReviewDecision(
                outcome=ReviewOutcome.REJECT,
                approver_id="system",
                reason="timeout after 48 hours",
            )

        decision = self._decision
        await workflow.execute_activity(
            record_decision, args=[spec, decision], **_ACT_CFG
        )

        if decision.outcome == ReviewOutcome.REJECT:
            return  # 拒絕就結束，不套用價格

        await workflow.execute_activity(apply_new_price, spec, **_ACT_CFG)

    @workflow.update
    async def update(self, decision: ReviewDecision) -> None:
        if self._decision is None:
            self._decision = decision  # 只接受第一次審核
wait_condition 的運作機制
wait_condition(predicate, timeout) 會持續檢查 predicate 是否為 True。
它不是輪詢（polling），而是在 Workflow 狀態改變（收到 signal/update/activity 完成等）時自動重新檢查。
在等待期間，Workflow 不占用 Worker 資源——Temporal Server 持有這個等待狀態。

client_update.py — 外部送入審核

handle = client.get_workflow_handle_for(
    ProductChangeWorkflow.run,
    workflow_id=WORKFLOW_ID,
)
await handle.execute_update(
    ProductChangeWorkflow.update,
    ReviewDecision(
        outcome=ReviewOutcome.APPROVE,
        approver_id="marketing_lead",
    ),
)
Update vs Signal 的選擇
UpdateSignal
回傳值可以有回傳值（等待 handler 完成）fire-and-forget，沒有回傳值
等待行為execute_update 會等到 handler 執行完畢signal 送出後立即 return
錯誤處理handler 中的 Exception 會傳回給呼叫端呼叫端無法得知處理結果
適用場景需要知道處理結果的互動（審批、RPC 風格）只需通知，不需等結果（事件觸發）

注意事項

Update handler 的冪等設計
if self._decision is None: 確保只接受第一次審核決定。如果沒有這個檢查，重複送 Update 可能導致覆蓋先前的審核結果。
TimeoutError 的處理
wait_condition 的 timeout 拋出的是 Python 標準的 TimeoutError，不是 Temporal 特有的 Exception。
Temporal 的 timer 是持久化的——即使 Worker 重啟，48 小時後仍然會觸發 timeout。這和 asyncio.sleep 在 Workflow 中的行為一致。
實務考量：審核通知
這個範例中 notify_reviewer 只是 print。在實際系統中，你會在這裡發 Slack 訊息、Email 或推播通知，並附上一個連結讓審核者可以觸發 Update（例如透過你的 API server 呼叫 execute_update）。

延伸練習

  1. 讓 Update handler 回傳一個 str（例如 "Accepted"），然後在 client_update.py 中印出回傳值
  2. 加入 Update validator：拒絕 approver_id 為空的審核決定
  3. 改成 Signal 版本，比較呼叫端的行為差異
  4. 加入「48 小時前 24 小時自動催促」的邏輯（提示：可以在 wait_condition 前先等 24 小時）
