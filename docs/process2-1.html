Process 2-1 — 商品上架流程（直接補償清單）

CompensationWorkflow Control FlowError Handling

學習目標

  • 理解補償機制：每一步執行前先記住對應的安全回滾操作，失敗時逆序執行補償 Activity
  • 學會直接在 Workflow 中維護補償清單，不依賴自製 helper
  • 理解補償順序為什麼通常要採用 LIFO（後完成的步驟先回滾）
  • 學會在 Workflow 中優雅處理部分成功的場景

流程概念

正常路徑（Happy Path）：

建立商品 → 設定價格 → 上架到店面 → 完成

失敗路徑（例如第 3 步失敗）：

建立商品 ✓ → 設定價格 ✓ → 上架失敗 ✗ → 補償 → 確認下架 → 還原價格 → 刪除商品

為什麼 Workflow 狀態是 Completed 不是 Failed？
因為 Workflow 在失敗時仍然 return ProductPublishResult(published=False, ...)，有回傳值就是 Completed。
若要讓 Workflow 變成 Failed 狀態，需要讓 exception 往上冒泡而不是 catch 住。兩種做法都合理，取決於業務需求。

程式碼解說

閱讀順序：models → activities → workflow → worker → starter
  1. models.py：ProductInput 與 ProductPublishResult 定義流程輸入與結果。
  2. activities.py：列出正向操作與補償操作。
  3. workflow.py：ProductOnboardingWorkflow.run 直接維護補償清單。
  4. worker.py：註冊商品上架 Workflow 與六個 Activity。
  5. starter.py：送出商品上架請求。

activities.py — Activity 清單

Activity用途
create_or_update_product建立或更新商品主資料。
setup_pricing_and_availability設定價格與可售狀態。
publish_to_storefront將商品上架到店面，可用 FAIL_PUBLISH 模擬失敗。
delete_product補償 create_or_update_product。
revert_pricing_and_availability補償價格與庫存設定。
unpublish_from_storefront補償店面上架。

這個教學版的 Activity 範例只傳真正需要的 product_id。Workflow 仍然接收完整 ProductInput，但不把整包 input 傳給每個 Activity；真實系統如果需要商品名稱、價格、庫存等欄位，可以改成傳明確的 command/data object。

workflow.py — 直接維護補償清單

# workflow.ActivityConfig 是 SDK 內建的 TypedDict，提供 activity 選項的型別檢查
_ACT_CFG: workflow.ActivityConfig = {
    "schedule_to_close_timeout": timedelta(seconds=15),
    "retry_policy": _RETRY,
}

compensations: list[Callable[[], Awaitable[None]]] = []
try:
    product_id = input.request_id
    compensations.append(
        lambda: workflow.execute_activity(delete_product, product_id, **_ACT_CFG)
    )
    product_id = await workflow.execute_activity(
        create_or_update_product, product_id, **_ACT_CFG
    )

    compensations.append(
        lambda: workflow.execute_activity(
            revert_pricing_and_availability, product_id, **_ACT_CFG
        )
    )
    await workflow.execute_activity(
        setup_pricing_and_availability, product_id, **_ACT_CFG
    )
    # ...
except Exception as e:
    for compensate in reversed(compensations):
        await compensate()
    return ProductPublishResult(published=False, message=f"...{e}")
這裡沒有使用自製 helper
Temporal 的核心概念是 Workflow 編排 Activity。補償也是 Activity，所以教材直接把補償清單寫在 Workflow 裡：正向 Activity 執行前先加入一個安全補償 callback，後面失敗時用 reversed(compensations) 反向呼叫。
先登記的前提
補償 Activity 必須是 idempotent / conditional：delete_product 遇到商品不存在要視為成功，revert_pricing_and_availability 遇到價格尚未設定要視為成功，unpublish_from_storefront 遇到尚未上架要視為成功。

activities.py — 正向操作 vs 補償操作

正向操作補償操作說明
create_or_update_productdelete_product建立 → 刪除
setup_pricing_and_availabilityrevert_pricing_and_availability設價格 → 還原價格
publish_to_storefrontunpublish_from_storefront上架 → 下架

補償要用反向順序執行。越晚完成的正向操作，通常越依賴前面的狀態，所以失敗時要先回滾最後一步，再逐步往前還原。

注意事項

補償機制 ≠ Transaction
補償機制提供的是「最終一致性」，不是 ACID 的原子性。在補償還沒執行完之前，系統處於「不一致」的中間狀態。
例如：商品已建立但尚未刪除的短暫時間窗口中，使用者可能看到一個半成品。
補償操作也可能失敗
這份範例直接在 Workflow 裡依序呼叫補償 Activity；如果補償 Activity 失敗，Workflow 會照一般 Activity 失敗規則進入 retry 或失敗處理。在實務上：
  • 補償操作本身也應該有 Retry Policy
  • 補償操作也應該是 idempotent 的
  • 如果補償也全部失敗，可能需要人工介入（記錄到 dead letter queue 或告警系統）
Workflow 回傳 Result 而不是拋出 Exception
這個 Workflow 在失敗時仍然回傳 ProductPublishResult(published=False, ...)，而不是讓 Exception 冒泡。
這表示 Workflow 的最終狀態是 Completed（不是 Failed），讓呼叫端可以從 result 判斷業務邏輯的成敗。
這是一個設計選擇：有些團隊偏好讓 Workflow Failed 來觸發告警，有些偏好用結果欄位來區分。

延伸練習

  1. 把補償迴圈改成順序錯誤的正向執行，觀察為什麼必須先還原價格再刪商品
  2. 讓第 1 步（create_or_update_product）就失敗，確認仍會執行已先登記的安全補償 delete_product
  3. 讓 delete_product（補償操作）也拋出 Exception，觀察 Workflow 最終狀態與 Event History
