Process 9 — Workflow 版本管理

workflow.patched()VersioningDeterminism

學習目標

  • 理解為什麼 Workflow 需要版本管理：修改 Workflow 邏輯可能破壞正在執行的 Workflow
  • 學會使用 workflow.patched() 安全地演進 Workflow 邏輯
  • 理解 Non-Deterministic Error 的成因與解法

為什麼需要版本管理？

核心問題
Temporal 靠 Event History 的 replay 來恢復 Workflow 的狀態。
如果你修改了 Workflow 的邏輯（例如新增一個 Activity），舊的 Event History 和新的程式碼對不上，Temporal 會拋出 Non-Deterministic Error。

範例：
  • v1：Activity A → Activity B
  • v2：Activity A → Activity C → Activity B
v1 的 Event History 中沒有 Activity C 的事件，但 v2 的程式碼期望它存在 → 爆炸。

workflow.patched() 的運作原理

if workflow.patched("greet-change-v2"):
    # 新邏輯
else:
    # 舊邏輯
workflow.patched() 做了什麼？
  • 新的 Workflow Execution（首次執行）：patched() 回傳 True，並在 Event History 中記錄一個 Marker 事件
  • Replay 舊的 Workflow Execution：檢查 History 中有沒有這個 patch 的 Marker
    • 有 Marker → 回傳 True（走新邏輯）
    • 沒有 Marker → 回傳 False（走舊邏輯）
這樣同一份程式碼就能同時相容新舊版本的 Workflow Execution。

版本演進的過程

# === 階段 1：最初的程式碼 ===
async def run(self, name: str) -> str:
    return f"Hello {name}"

# === 階段 2：加入 v1 ===
async def run(self, name: str) -> str:
    if workflow.patched("greet-change-v1"):
        return f"Hi {name}"       # 新的 Workflow 走這
    else:
        return f"Hello {name}"    # 舊的 Workflow replay 走這

# === 階段 3：加入 v2 ===
async def run(self, name: str) -> str:
    if workflow.patched("greet-change-v2"):
        return f"Hey {name}"      # 最新
    elif workflow.patched("greet-change-v1"):
        return f"Hi {name}"       # v1
    else:
        return f"Hello {name}"    # 原始

# === 階段 4：確認所有舊 Workflow 都已完成後，清理 ===
async def run(self, name: str) -> str:
    return f"Hey {name}"          # 安全移除所有 patched() 分支

注意事項

什麼時候可以移除 patched() 分支？
只有在確認所有使用舊邏輯的 Workflow Execution 都已經完成（Completed / Failed / Cancelled / Terminated）之後，才能安全移除舊的分支。
如果還有舊的 Workflow 在 Running 狀態，移除舊分支會導致那些 Workflow 的 replay 失敗。
patched() 的替代方案
方案適用場景說明
workflow.patched()小幅修改，需要向後相容同一份程式碼支援新舊邏輯
Worker Versioning大幅重構用不同的 Task Queue 或 Build ID 隔離新舊 Worker
新的 Workflow Type完全不同的流程定義新的 Workflow class，不影響舊的
Patch ID 命名
Patch ID（例如 "greet-change-v2"）是一個自定義字串。建議有意義的命名，因為它會出現在 Event History 的 Marker 中，方便日後除錯。

延伸練習

  1. 把 return f"Hey {name}" 改成呼叫一個 Activity（例如查資料庫取得最新的 greeting 模板），體會為什麼這類修改特別需要 patched()
  2. 嘗試在不用 patched() 的情況下修改一個正在執行的 Workflow 的邏輯，觀察 Non-Deterministic Error 長什麼樣子
  3. 用 workflow.deprecate_patch() 標記一個 patch 為已棄用，這是清理路徑中的中間步驟
