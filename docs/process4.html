程式碼解說

閱讀順序：models → activities → workflow → worker → starter
  1. models.py：LongJobSpec 定義資料總量。
  2. activities.py：長時間處理資料並定期 heartbeat。
  3. workflow.py：設定 Activity timeout、heartbeat timeout 與 retry。
  4. worker.py：註冊長任務 Workflow 與 Activity。
  5. starter.py：啟動長時間任務。

activities.py — Activity 清單

Activity用途
process_dataset處理資料集、送出 heartbeat，並在重試時從 heartbeat details 續跑。

workflow.py — Timeout 配置

last = await workflow.execute_activity(
    process_dataset,
    spec,
    start_to_close_timeout=timedelta(minutes=5),   # 單次執行上限 5 分鐘
    heartbeat_timeout=timedelta(seconds=30),        # 30 秒沒收到心跳就判定死亡
    retry_policy=_RETRY,                            # 最多重試 10 次
)
heartbeat_timeout 的意義
Temporal Server 期望 Activity 每 30 秒至少送一次心跳。如果超過 30 秒沒收到：
  1. Temporal 判定 Worker 可能掛了
  2. 將 Activity 標記為 timeout
  3. 根據 Retry Policy 將 Activity 重新分派給其他（或同一個）Worker
  4. 新的 Activity 可以從 heartbeat_details 取得上次的進度

activities.py — 心跳與斷點續傳

@activity.defn
async def process_dataset(spec: LongJobSpec) -> int:
    info = activity.info()
    # 從 heartbeat_details 取得上次的進度（如果有的話）
    start = int(info.heartbeat_details[0]) if info.heartbeat_details else 0

    for i in range(start, spec.total):   # 從上次的進度繼續
        await asyncio.sleep(0.5)
        activity.heartbeat(i)            # 送出心跳，附帶當前進度
        print(f"progress: {i} %")

    return spec.total - 1
斷點續傳的關鍵
activity.heartbeat(i) 的參數 i 就是 heartbeat details。當 Activity 被重試時，Temporal 會把最後一次成功送達的 heartbeat details 放在 activity.info().heartbeat_details 中。

heartbeat details 可以是任何可序列化的值：一個數字、一個 dict、一個 dataclass 都行。

注意事項

心跳頻率的設計
  • 心跳太頻繁（例如每 10ms 送一次）會造成 Temporal Server 大量負載
  • 心跳太稀疏（接近 heartbeat_timeout）會導致正常工作的 Activity 被誤判死亡
  • 經驗法則：心跳間隔 ≤ heartbeat_timeout 的 1/3。本範例 timeout 30 秒，每 0.5 秒送一次心跳是合理的。
SDK 內建的心跳節流（Throttling）
Temporal Python SDK 會自動節流心跳：如果你在短時間內送出大量 heartbeat，SDK 只會實際傳送一部分到 Server。所以即使你在迴圈每次迭代都呼叫 activity.heartbeat()，也不會造成網路風暴。

但 heartbeat_details 的值只有在心跳實際傳送到 Server 時才會被記錄。因此斷點續傳的精確度取決於 SDK 節流後實際送出的頻率。
不需要 Heartbeat 的場景
如果 Activity 很短（幾秒內完成），不需要設 heartbeat。Heartbeat 主要用於：
  • 處理大批資料（需要斷點續傳）
  • 長時間等待外部系統回應（需要偵測 Worker 死亡）
  • 需要讓外部系統查詢進度

延伸練習
