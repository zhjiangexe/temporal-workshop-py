Process 6 — Workflow 取消

CancellationCancelledErrorCleanup

學習目標

  • 理解 Temporal 的取消機制：外部發送取消請求 → Workflow 收到 CancelledError
  • 學會在取消時做清理工作（cleanup）
  • 理解取消請求是「協作式」的——Workflow 可以決定如何回應

流程概念

程式碼解說

閱讀順序：models → activities → workflow → worker → starter
  1. models.py：本流程未使用額外資料模型。
  2. activities.py：本流程沒有 Activity，重點在 Workflow 取消。
  3. workflow.py：MyWorkflow.run 示範收到 CancelledError 後 cleanup 並 re-raise。
  4. worker.py：註冊取消示範 Workflow。
  5. client_cancel.py：扮演 starter，啟動 Workflow 後送出取消請求。

activities.py — Activity 清單

Activity用途
本流程沒有 Activity。

workflow.py — 處理取消

@workflow.defn
class MyWorkflow:
    @workflow.run
    async def run(self, s: str) -> str:
        try:
            print("scope one")
            await asyncio.sleep(7200)     # 等待 2 小時
        except asyncio.CancelledError:
            print("scope two")            # 清理工作
            print("out")
            raise                         # 必須 re-raise！
        return "zero"
必須 re-raise CancelledError
如果你 catch 了 CancelledError 但不 re-raise，Workflow 會繼續執行而不是被取消。
在 try/except 中做完清理工作後，一定要 raise 把 CancelledError 傳遞出去，讓 Temporal 正確標記 Workflow 為 Cancelled 狀態。

client_cancel.py — 發送取消請求

handle = await client.start_workflow(
    MyWorkflow.run,
    "payload",
    id="wf-cancel-demo-1",
    task_queue=TASK_QUEUE,
)
await handle.cancel()      # 發送取消請求
print("Cancel request sent.")
cancel() 是非同步的
handle.cancel() 只是發送取消請求，不代表 Workflow 立即結束。
Workflow 會在下一個 await 點收到 CancelledError。如果 Workflow 正在執行一段不含 await 的同步程式碼，取消會被延遲到下一個 await。

注意事項

取消是「協作式」的（Cooperative Cancellation）
Temporal 不會強制殺掉 Workflow。取消請求被轉換成 CancelledError 拋進 Workflow 的 async 執行流中。
Workflow 可以：
  • 接受取消：catch → cleanup → re-raise（本範例的做法）
  • 忽略取消：catch → 不 raise → 繼續執行（不推薦，但在某些場景有用）
  • 部分取消：catch → 完成必要的步驟 → 再 raise
取消傳播到 Activity
如果 Workflow 被取消時正在執行一個 Activity：
  • Workflow 層會收到 CancelledError
  • Activity 本身會收到 cancellation（透過 activity.is_cancelled() 或心跳機制）
  • Activity 的取消也是協作式的——Activity 需要自行檢查並退出
也可以從 Web UI 取消
在 Workflow 詳情頁的右上角有 "Terminate" 和 "Cancel" 按鈕。
  • Cancel：發送取消請求（協作式，Workflow 可以做 cleanup）
  • Terminate：強制終止（Workflow 沒有機會做 cleanup）

延伸練習

  1. 在 catch 區塊中加入一個 Activity（例如寄取消通知 email），看看取消時是否能執行 Activity
  2. 嘗試不 re-raise CancelledError，觀察 Workflow 的最終狀態是 Completed 而不是 Cancelled
  3. 用 Web UI 的 Terminate 功能終止一個 Workflow，比較 Terminate 和 Cancel 在 Event History 中的差異
