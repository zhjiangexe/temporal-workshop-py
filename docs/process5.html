Process 5 — 庫存管理實體（Entity Pattern）

Entity PatternSignalUpdateQueryContinue-As-New

學習目標

  • 理解 Entity Workflow：長期存活的有狀態 Workflow，等同於一個持久化的「物件」
  • 區分三種外部互動方式：Signal（非同步）、Update（同步）、Query（唯讀）
  • 學會 continue_as_new 防止 Event History 無限增長
  • 理解 inbox pattern：用佇列收集命令，在主迴圈中依序處理

流程概念

流程圖區塊可拖曳右下角放大，較長的圖也可水平捲動。

┌─────────────────────────────────────────────────────┐
│  InventoryEntityWorkflow（長期存活）                    │
│                                                     │
│  while True:                                        │
│    等待 inbox 有命令 OR history 太長需要 continue-as-new  │
│    處理 inbox 中的所有命令（RESTOCK / PURCHASE）         │
│    如果 history 太長 → continue_as_new(保留當前狀態)     │
│                                                     │
│  外部介面：                                           │
│    signal submit(command)  → 非同步送命令               │
│    update invoke(command)  → 同步送命令，等結果回傳      │
│    query  get_stock()      → 讀取當前庫存               │
└─────────────────────────────────────────────────────┘

程式碼解說

閱讀順序：models → activities → workflow → worker → starter
  1. models.py：Command、CommandType、CommandResult 定義庫存命令。
  2. activities.py：真正修改庫存並處理業務錯誤。
  3. workflow.py：Entity Workflow 維護 inbox、狀態與 continue-as-new。
  4. worker.py：註冊庫存 Workflow 與 Activity instance。
  5. starter.py：啟動庫存實體並示範 Signal、Update、Query。

activities.py — Activity 清單

Activity用途
restock補貨並回傳新的庫存數量。
purchase扣庫存；庫存不足時拋出不可重試的業務錯誤。

workflow.py — 主迴圈

@workflow.run
async def run(self, product_id: str, initial_stock: int = 0) -> None:
    self._product_id = product_id
    if initial_stock > 0:
        self._cached_stock = await workflow.execute_activity_method(
            InventoryActivities.restock,
            args=[product_id, initial_stock, f"INIT-{product_id}"],
            **_ACT_CFG,
        )

    while True:
        # 等待條件：inbox 有東西，或 Temporal 建議 continue-as-new
        await workflow.wait_condition(
            lambda: self._inbox
            or workflow.info().is_continue_as_new_suggested
        )

        # 處理所有待處理的命令
        while self._inbox:
            command = self._inbox.popleft()
            result = await self._process_command(command)
            self._results[command.op_key] = result

        # History 太長，做 continue-as-new
        if workflow.info().is_continue_as_new_suggested:
            await workflow.wait_condition(workflow.all_handlers_finished)
            workflow.continue_as_new(args=[self._product_id, self._cached_stock])
為什麼需要 continue_as_new？
Temporal 的 Event History 會隨著每個 Activity、Signal、Update 不斷增長。
當 History 太大時（預設建議門檻約 2000 個事件）：
  • Replay 時間變長（Worker 重啟後要 replay 整個 history）
  • 佔用更多 Temporal Server 的儲存空間
continue_as_new 會結束當前 Workflow 並立即啟動一個全新的 Workflow Execution（保持相同的 Workflow ID），帶著你給的參數重新開始，Event History 歸零。

workflow.info().is_continue_as_new_suggested 是 Temporal 內建的建議信號。

三種互動方式比較

# Signal — 非同步，fire-and-forget
@workflow.signal
def submit(self, command: Command) -> None:
    self._inbox.append(command)

# Update — 同步，等待結果
@workflow.update
async def invoke(self, command: Command) -> CommandResult:
    self._inbox.append(command)
    await workflow.wait_condition(lambda: command.op_key in self._results)
    return self._results.pop(command.op_key)

# Query — 唯讀，不改變 Workflow 狀態
@workflow.query
def get_stock(self) -> int:
    return self._cached_stock

SignalUpdateQuery
方向送進送進 + 等回覆讀出
改變狀態可以可以不可以
回傳值無有有
記錄在 History是是否
適用場景通知、觸發RPC、需要確認的操作查詢狀態

Inbox Pattern

為什麼不直接在 Signal/Update handler 中執行 Activity？
Signal handler 和 Update handler 在 Temporal 中有特殊的執行語義。把所有命令先收到 inbox（deque），再在主迴圈中依序處理，有幾個好處：
  • 序列化處理：避免併發修改 _cached_stock 導致的 race condition
  • 統一的控制流：continue-as-new 的判斷只需要在一個地方
  • 可預測的 replay：所有 Activity 都在 @workflow.run 的上下文中執行

Continue-As-New 前的安全檢查

if workflow.info().is_continue_as_new_suggested:
    await workflow.wait_condition(workflow.all_handlers_finished)  # 等所有 handler 完成
    workflow.continue_as_new(args=[self._product_id, self._cached_stock])
all_handlers_finished 很重要！
如果在 Update handler 還在等待結果時就 continue-as-new，那個 Update 呼叫會收到錯誤。
workflow.all_handlers_finished 確保所有正在進行的 Update handler 都已經完成後，才執行 continue-as-new。

非可重試的錯誤

_ACT_CFG: workflow.ActivityConfig = {
    "retry_policy": RetryPolicy(
        non_retryable_error_types=["InvalidInput", "StockError"],
    ),
}

當 Activity 拋出 ApplicationError(type="InvalidInput") 時，Temporal 不會重試——因為業務邏輯錯誤重試也不會成功。

注意事項

Entity Workflow 的 Workflow ID 設計
id=f"inventory:{product_id}" 用商品 ID 作為 Workflow ID，確保每個商品只有一個 Entity Workflow。
如果嘗試用相同 ID 啟動第二個 Workflow，Temporal 會拒絕（WorkflowExecutionAlreadyStarted）。
這就是 Entity 模式的核心：Workflow ID 就是實體的 primary key。
continue_as_new 後的 Workflow ID
continue-as-new 後，Workflow ID 保持不變。外部客戶端可以繼續用相同的 ID 發送 Signal/Update/Query。
對外部來說，continue-as-new 是透明的。
