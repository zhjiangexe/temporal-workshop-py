workflow.py — 最精簡的 Signal + Wait

@workflow.defn
class AwaitWorkflow:
    def __init__(self):
        self._ready = False                          # 旗標

    @workflow.run
    async def run(self) -> str:
        try:
            await workflow.wait_condition(
                lambda: self._ready,                 # 等這個條件變 True
                timeout=timedelta(hours=1),          # 或 1 小時後 timeout
            )
            return "READY"
        except TimeoutError:
            return "TIMEOUT"

    @workflow.signal
    def mark_ready(self) -> None:
        self._ready = True                           # Signal 修改旗標
wait_condition 的語義
wait_condition(predicate, timeout) 等同於：「幫我等到 predicate() 回傳 True，但最多等 timeout。」

Temporal 不是用輪詢來檢查 predicate 的。當 Workflow 收到任何事件（signal、update、activity 完成等），SDK 會自動重新評估所有 wait_condition 的 predicate。
Signal 可以在 Workflow 啟動前就送出
如果你在 Workflow 還沒被 Worker 拉取之前就送了 Signal，Signal 會被 Temporal Server 暫存。等 Worker 開始執行 Workflow 時，Signal 已經在等著了。
這表示 wait_condition 可能在第一次檢查時就發現條件已成立，立即通過。

Signal vs Update 選擇指南

用 Signal 的場景
  • 呼叫端不需要知道處理結果（fire-and-forget）
  • 單純通知/觸發：「某件事發生了」
  • 需要 buffering：Signal 可以在 Workflow 未執行時累積
  • 高吞吐量：Signal 比 Update 輕量
用 Update 的場景（見 Process 3, 7）
  • 呼叫端需要等待回覆（RPC 風格）
  • 需要知道操作是否成功
  • 需要回傳計算結果

注意事項

Signal handler 中不能有 await
Signal handler 應該盡量簡短：修改一個 flag、往 deque 塞一筆資料。不要在 Signal handler 中呼叫 Activity 或做長時間操作。
需要在收到 Signal 後執行複雜邏輯？在 @workflow.run 的主流程中用 wait_condition 等待 Signal 修改的狀態，然後在主流程中做處理（見 Process 5 的 inbox pattern）。
多個 Signal 的處理
如果在 wait_condition 通過之前連續送了多個 Signal，所有 Signal handler 都會被執行，但 wait_condition 只會在條件成立時通過一次。
如果你需要處理每個 Signal，應該用 queue（deque）來收集，而不是用 boolean flag。

延伸練習

  1. 加入一個帶參數的 Signal：mark_ready(reason: str)，讓 Workflow 回傳 f"READY: {reason}"
  2. 加入多重條件：需要收到兩個不同的 Signal（approval_a + approval_b）才通過
  3. 改成用 asyncio.sleep + flag 而不是 wait_condition，比較兩者的行為差異（提示：sleep 不會因為 flag 改變而提前結束）
