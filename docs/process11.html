Process 11 — 非同步 Activity 完成

Async CompletionTask TokenExternal System

學習目標

  • 理解 Async Activity Completion：Activity 函式 return 後，Activity 還沒算「完成」
  • 學會用 task_token 在外部（另一個程序/服務）完成 Activity
  • 認識適用場景：等待外部系統回覆（webhook、人工操作、第三方 API callback）

與一般 Activity 的差異

一般 ActivityAsync Activity
完成方式函式 return 即完成函式 return 後仍在等待，由外部用 task_token 完成
Worker 佔用函式執行期間佔用 Worker slotraise_complete_async() 後立即釋放 Worker slot
適用場景可在 Worker 內完成的工作需要等待外部系統回應的工作

程式碼解說

閱讀順序：models → activities → workflow → worker → starter
  1. models.py：本流程未使用額外資料模型。
  2. activities.py：Activity 取得 task token，交由外部完成。
  3. workflow.py：AsyncWf.run 以一般方式等待 Activity 結果。
  4. worker.py：本流程未提供獨立 worker 檔，測試中會註冊 AsyncActImpl。
  5. starter.py：本流程未提供獨立 starter，可用測試互動。

activities.py — Activity 清單

Activity用途
wait_external取得 task token 後呼叫 raise_complete_async()，等待外部完成。

activities.py — Activity 端

class AsyncActImpl:
    def __init__(self, client: Client):
        self._client = client               # 需要 Client 來完成 Activity

    @activity.defn
    async def wait_external(self, id: str) -> str:
        token = activity.info().task_token   # 取得 task token
        asyncio.create_task(self._complete_later(token, id))  # 啟動背景任務
        activity.raise_complete_async()      # 告訴 Temporal：我先 return，但 Activity 還沒完成

    async def _complete_later(self, token: bytes, id: str) -> None:
        handle = self._client.get_async_activity_handle(task_token=token)
        await handle.complete(f"READY-{id}")   # 在外部完成 Activity
raise_complete_async() 的意義
呼叫 activity.raise_complete_async() 後：
  • Activity 函式立即結束
  • Worker 釋放這個 Activity 的執行 slot（可以處理其他 Activity）
  • 但 Temporal Server 仍然認為 Activity 在「進行中」
  • 直到有人用 task_token 呼叫 handle.complete() 或 handle.fail()

workflow.py — Workflow 端完全不知道差異

@workflow.defn
class AsyncWf:
    @workflow.run
    async def run(self, id: str) -> str:
        return await workflow.execute_activity_method(
            AsyncActImpl.wait_external,
            id,
            start_to_close_timeout=timedelta(seconds=10),
        )

Workflow 端的程式碼完全一樣——它不知道也不需要知道 Activity 是同步還是非同步完成的。await 會等到 Activity 真正完成。

Task Token 是什麼？

Task Token
Task Token 是一個 opaque 的 bytes 值，由 Temporal Server 產生。它唯一識別一個 Activity Task。
誰持有這個 token，誰就能完成（或失敗）這個 Activity。

在實際場景中，你會把 task token：
  • 儲存到資料庫，等 webhook 收到回覆後用 token 完成 Activity
  • 傳給另一個微服務，讓它處理完後回報
  • 放在一個 UI 上，等人工操作後完成

實際應用場景

場景 1：等待第三方支付確認 
# Activity 中
token = activity.info().task_token
save_to_db(payment_id, token)       # 儲存 token
call_payment_api(payment_id)        # 發起支付
activity.raise_complete_async()     # 不等結果

# Webhook handler 中（收到支付回覆時）
token = load_from_db(payment_id)
handle = client.get_async_activity_handle(task_token=token)
await handle.complete(payment_result)
場景 2：等待人工審核 
# Activity 中
token = activity.info().task_token
create_review_task(token, document_id)  # 建立審核任務
activity.raise_complete_async()

# 審核系統 API 中（審核者按下「核准」時）
token = get_token_from_review_task(task_id)
handle = client.get_async_activity_handle(task_token=token)
await handle.complete(ReviewResult(approved=True))

注意事項

start_to_close_timeout 仍然生效
即使用了 async completion，start_to_close_timeout 仍然會計時。如果在 timeout 內沒有人呼叫 handle.complete()，Activity 會被判定為 timeout 並觸發 retry。
等待外部系統回覆可能需要很長時間，記得設定夠長的 timeout，或者使用 heartbeat 來延長 Activity 的存活時間。
Task Token 的安全性
Task Token 等同於「完成 Activity 的權限」。持有 token 的任何人都可以完成或失敗該 Activity。
不要將 token 暴露在公開的 URL 或 log 中。
handle.fail() 讓 Activity 失敗
除了 handle.complete(result)，也可以用 handle.fail(error) 讓 Activity 以失敗狀態結束。
還有 handle.heartbeat(details) 可以從外部送心跳（雖然 Activity 函式已經 return）。

延伸練習

  1. 加入延遲：讓 _complete_later 先 await asyncio.sleep(3) 再 complete，模擬外部系統的處理時間，觀察 Web UI 中 Activity 的等待狀態
  2. 使用 handle.fail() 替代 handle.complete()，觀察 Activity 失敗後是否觸發 retry
  3. 把 _complete_later 改為一個獨立的 script（模擬完全獨立的外部程序），用硬編碼的 task token 來完成 Activity
