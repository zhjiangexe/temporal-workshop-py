程式碼解說

閱讀順序：models → activities → workflow → worker → starter
  1. models.py：RegisterRequest 定義註冊輸入資料。
  2. activities.py：所有會碰外部系統或需要重試的工作。
  3. workflow.py：RegistrationWorkflow.register 串起三個 Activity。
  4. worker.py：註冊 Workflow 與 Activity 到 registration-tq。
  5. starter.py：建立 Client 並啟動一筆註冊 Workflow。

activities.py — Activity 清單

Activity用途
create_account建立使用者帳號，並用 user_id 避免重複建立。
add_points新增註冊點數，展示 RetryPolicy 與 idempotency key。
send_email寄送歡迎信，使用 message_id 避免重複寄送。

models.py — 資料結構

@dataclass
class RegisterRequest:
    user_id: str
    email: str

用 @dataclass 定義 Workflow 的輸入。Temporal Python SDK 會自動序列化 dataclass。

workflow.py — Step 1：設定重試策略

_RETRY = RetryPolicy(
    maximum_attempts=5,

    initial_interval=timedelta(seconds=5),
    backoff_coefficient=2,
    maximum_interval=timedelta(seconds=100),
)
  1. maximum_attempts=5 表示 Activity 最多嘗試 5 次。
  2. initial_interval 是第一次重試前要等多久。
  3. backoff_coefficient=2 讓等待時間每次加倍：5s → 10s → 20s → 40s。
  4. maximum_interval 限制單次重試等待時間不要無限變長。

workflow.py — Step 2：共用 Activity 參數

_ACT_CFG: workflow.ActivityConfig = {
    "schedule_to_close_timeout": timedelta(seconds=120),

    "start_to_close_timeout": timedelta(seconds=15),

    "heartbeat_timeout": timedelta(seconds=10),

    "retry_policy": _RETRY,
}
Timeout 三兄弟
schedule_to_close：Activity 從被排入佇列到完成的總時間（含所有重試）。
start_to_close：Activity 單次執行時間上限。
heartbeat_timeout：Activity 必須在此時間內送出至少一次 heartbeat，否則視為 worker 掛了，Temporal 會重新分派。

這些設定放進 _ACT_CFG，後面每個 Activity 都可以用 **_ACT_CFG 套用同一組規則。

workflow.py — Step 3：定義 Workflow 入口

@workflow.defn
class RegistrationWorkflow:

    @workflow.run
    async def register(self, request: RegisterRequest) -> None:
        ...
  1. @workflow.defn 把 class 註冊成 Temporal Workflow 定義。
  2. @workflow.run 標記真正會被啟動的入口方法。
  3. request 是 Starter 傳進來的輸入，也就是前面的 RegisterRequest。

workflow.py — Step 4：建立帳號

await workflow.execute_activity_method(
    RegistrationActivities.create_account,
    request,
    **_ACT_CFG,
)

第一個 Activity 先建立使用者帳號。因為這個 Activity 是 class method，所以使用 execute_activity_method。這裡直接把整個 request 傳進去。

workflow.py — Step 5：新增註冊點數

await workflow.execute_activity_method(
    RegistrationActivities.add_points,
    args=[
        request.user_id,
        500,
        "signupBonus",
    ],
    **_ACT_CFG,
)
  1. 這一步等建立帳號完成後才會執行，因為前一行有 await。
  2. args 依序傳入 user_id、點數數量、以及 idempotency 用的操作名稱。
  3. "signupBonus" 會成為 dedupe key 的一部分，避免重試時重複加點。

workflow.py — Step 6：寄歡迎信

await workflow.execute_activity_method(
    RegistrationActivities.send_email,
    args=[
        request.email,
        f"welcome-{request.user_id}",
    ],
    **_ACT_CFG,
)
Sequential Workflow 的核心
三個 Activity 都用 await 串起來，所以順序固定是：建立帳號 → 新增點數 → 寄歡迎信。

如果其中一步失敗，Temporal 會先依照 _RETRY 重試；重試仍失敗時，後面的 Activity 不會繼續執行。

activities.py — 冪等設計

class RegistrationActivities:
    def __init__(self):
        self._users: set[str] = set()   # 追蹤已建立的帳號
        self._ops: set[str] = set()     # 追蹤已完成的操作

    @activity.defn
    async def add_points(self, user_id: str, points: int, op_key: str) -> None:
        dedupe_key = f"POINTS#{user_id}#{op_key}"
        if dedupe_key in self._ops:     # 已做過就跳過
            return
        self._ops.add(dedupe_key)
        await asyncio.sleep(5)          # 模擬 IO
為什麼 Activity 必須 idempotent？
Temporal 在 Activity 失敗（timeout、worker crash）時會自動重試。如果 Activity 不是 idempotent，重試可能導致重複建帳號、重複加點數、重複寄信。

常見做法：用 dedupe key 檢查是否已執行過。在實際專案中，這個 dedupe 機制通常由資料庫的 unique constraint 或外部服務的 idempotency key 來保證。

worker.py — Worker 配置

worker = Worker(
    client,
    task_queue=TASK_QUEUE,
    workflows=[RegistrationWorkflow],
    activities=[acts.create_account, acts.add_points, acts.send_email],
    max_concurrent_activities=50,           # 同時處理的 Activity 上限
    max_concurrent_workflow_tasks=20,       # 同時處理的 Workflow Task 上限
    max_activities_per_second=50.0,         # Activity 速率限制
    max_task_queue_activities_per_second=100.0,  # Task Queue 層級速率限制
)
Worker 調校參數
max_concurrent_activities：控制單一 Worker 的並行處理能力。設太高可能耗盡記憶體。
max_activities_per_second：保護下游服務，避免 Worker 發出過多請求。
這些都是單一 Worker 層級的設定。多個 Worker 共享同一個 Task Queue 時，各自獨立計算。

注意事項

Workflow 程式碼的約束（Determinism）
Workflow 的 @workflow.run 方法必須是確定性的（deterministic）：
  • 不能用 time.time()、random.random() → 用 workflow.time()、workflow.random()
  • 不能做 IO（HTTP、DB）→ 所有 IO 放在 Activity 裡
  • 不能用 threading → 用 asyncio
  • import non-deterministic 模組要包在 workflow.unsafe.imports_passed_through()
Task Queue 命名
Worker 和 Starter 必須使用相同的 Task Queue 名稱。如果名稱不一致，Workflow 會一直等待但永遠沒有 Worker 來處理。在 Web UI 上會看到 Workflow 停在 Running 狀態不動。
